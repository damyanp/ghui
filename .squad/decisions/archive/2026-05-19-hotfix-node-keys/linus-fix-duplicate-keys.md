## Root cause

`RecipeNodeBuilder::push_item` (`ghui-app/src/nodes/recipe_builder.rs:567` pre-fix) set `Node.id = id.0.clone()` — the bare `WorkItemId`. With `Pivot(Epic) + Hierarchy + show_ghost_ancestors`, the same work item legitimately appears in **multiple** Epic buckets — once as a real item in its own Epic bucket, and once as a **ghost ancestor** in every other Epic bucket whose items descend from it. Two `Node` entries sharing an id → Svelte's `{#each rows as row (row.id)}` aborts → expand/collapse + drag tracking + visibility set (all keyed on `row.id`) silently die for everything past the first duplicate.

Group nodes already had a path-prefixed id (`group_node_id`, recipe_builder.rs:602). Work-item nodes didn't. They do now.

## Design lessons worth flagging

1. **When the platform supplies a "key invariant", assert the invariant directly.** Svelte `{#each}` requires unique keys. The right regression test isn't "the snapshot looks correct"; it's `HashSet::from_iter(nodes.iter().map(|n| &n.id)).len() == nodes.len()`. The new test does exactly that, plus asserts the parent's dual occurrence (real + ghost) is preserved. If PR #76 had included this test, this bug wouldn't have shipped. Pattern adopted going forward.

2. **Test helpers are part of the contract.** `render_recipe_nodes` formats by `work_item_id.0`, not `node.id`. That meant flipping `Node.id`'s representation left every existing preset snapshot stable — pure good fortune from a past design choice. Worth being deliberate about: when writing a snapshot helper that summarises *some* fields, choose the ones whose semantics you want to pin, knowing that fields not in the format become free to refactor without snapshot churn.

3. **Multiple callsites had been using `node.id` as a `WorkItemId` for lookup** (`workItems[node.id]`, `itemUpdateBatcher.add(node.id, …)`, `Change.workItemId = draggedRowId`). They all worked **by accident** pre-fix because `Node.id` was the `WorkItemId`. After the fix they require `node.data.workItemId` (with `node.data.type === "workItem"` narrowing). TS found them all once the ts-rs binding regenerated — `npm run check` was the source of truth. The `TreeTable.svelte` `Row<T>` generic constraint on `GhostAwareNodeData` keeps the narrowing type-safe without escape hatches. Avoid having two ids whose values incidentally coincide — that's the bug that bit us.

4. **Anti-pattern declined:** It would have been tempting to "fix" this by suppressing ghost rendering, but ghost ancestry is the intentional UX (Task 7, PR #76). The fix is in the key, not the data.

5. **Anti-pattern declined:** Recovering the WorkItemId by parsing the path-prefixed `Node.id` on the frontend would be brittle — WorkItemIds can contain `/`. `node.data.workItemId` is the only correct source.

## Out of scope — recommended follow-ups for Rusty / Basher

- **Render-key uniqueness invariant could be asserted at the type level.** Currently it's a runtime invariant maintained by `child_path`. A newtype like `RenderKey(String)` constructed only through `child_path` would make it harder to regress. Probably premature; flagging anyway.
- **The drag/drop handler in `WorkItemTree.svelte` silently dropped the `setParent` case when the target was a group AND the dragged was a group** (now narrowed with explicit `else if targetNode.data.type === "workItem"`). Previously the `else` branch would have fired for group targets too; harmless because the drag UI doesn't allow group→group drops, but worth tightening as I did here.
