## Learnings

<!-- Append new learnings below. Each entry is something lasting about the project. -->

- 📌 Svelte 5 runes, NOT Svelte 4 stores. `$state`, `$derived`, `$bindable`, `$props`.
- 📌 Context pattern: `setWorkItemContext()` in root `+page.svelte`, consumed in children. WorkItemContext lives in `app/src/lib/WorkItemContext.svelte.ts`.
- 📌 Vitest tests live next to source as `*.test.ts`. Plain TypeScript, no Svelte/Tauri runtime imports — `invoke` from `@tauri-apps/api/core` is NOT available in tests.
- 📌 Testable logic pattern: extract from `.svelte.ts` class → sibling pure module → class delegates. Example: `WorkItemContext` consumes `filterableFields.ts`.
- 📌 Styling: Tailwind 4 + Skeleton (Cerberus theme). Reuse tokens like `bg-primary-100-900`, `bg-surface-300-700`. NEVER add CSS comments with specific hex values — they go stale.
- 📌 Drag interactions: Pointer Events with `setPointerCapture` / `releasePointerCapture`, element-level handlers — NOT document-level (leak risk).
- 📌 Toolbar: all buttons via `AppBarButton`, always visible, disabled when unavailable.
- 📌 Icons: `@lucide/svelte` for UI, `@primer/octicons` for GitHub-specific.
- 📌 Bindings dir: `app/src/lib/bindings/` (auto-generated by ts-rs on `cargo test --all`). Don't hand-edit.
- 📌 Pivoting Task 3 component API: `<RecipeBar bind:value={cfg} onApply={(cfg) => …} />`. Props: `value: PivotConfig`, `onApply: (cfg: PivotConfig) => void`.
- 📌 Phase 2 additive rule: NEVER modify `WorkItemTree.svelte` rendering logic — reserved for Task 6. RecipeBar is built and tested standalone in `/dev/recipe-bar` demo route, NOT mounted in main toolbar until Task 6.
- 📌 Team update (2026-05-19): PR #71 review revealed dead-stub UI toggles ("Show counts", "Collapse single-valued", "Hide closed items") that are disabled with no backing behavior. These violate the project rule: *"Don't add UI elements until they have working functionality."* Linus: these stubs belong to Task 9 (final polish). Reject them from Task 3; leave disabled UI for later phases.
- 📌 Async parser pattern (2026-05-19, PR #71 revision): `recipeParser.ts` (merged from PR #73) is the ONLY recipe parser — a thin async Tauri shim. When refactoring components from sync to async parsing: (1) use `$effect` with a cancellation guard (`let cancelled = false; return () => { cancelled = true; }`) for external value sync; (2) track the last-emitted recipe reference (plain `let`, not `$state`) to skip redundant re-formatting on self-emitted updates; (3) extract `applyText()` and `setToggle()` pure helpers into a sibling `.ts` module so tests can inject mock parse/format functions without mocking `invoke` directly.
- 📌 RecipeBar prop interface (2026-05-19, PR #71 / Task 6): `<RecipeBar bind:value={cfg} onApply={(cfg) => …} />` — props are `value: PivotConfig` (bindable) and `onApply: (cfg: PivotConfig) => void`. Mount pattern for toggled panels: `$state(false)` boolean + `{#if open}<div class="border-b border-surface-300-700 px-4 py-2">…</div>{/if}` placed immediately after `</AppBar>`, before any panel components (same as LogPanel pattern).

- 📌 Task 8 (Multi-value Explode toggle, 2026-05-19, PR #75): the toggle was already wired end-to-end by Tasks 1–6 — Rust `RecipeNodeBuilder::assignee_field_values` branches on Combined/Explode (recipe_builder.rs:478–512) with an existing snapshot test (`test_recipe_builder_multi_value_combined_vs_explode`); `ViewConfigCache` in `ghui-app/src/lib.rs` already persists `pivot_config` (Explode round-trip pinned by `test_view_config_cache_round_trip_pivot_config`); the Svelte checkbox + `setToggle` plumbing landed in PR #71. Task 8 became a coverage-only PR: extracted a pure `getToggleChecked(config, toggle)` helper into `recipeBarState.ts` (mirrors `setToggle`) so vitest can pin the three issue-spec cases — initial-state reflection, on→explode, off→combined — without touching a Svelte/Tauri runtime. Pattern reused from the Task 3 `applyText/setToggle` extraction. Always re-read the prerequisite code before assuming a "do the work" task isn't actually "the work is done, add the tests."
- 📌 Pre-existing local clippy noise (as of 2026-05-19): `cargo clippy --all -- -D warnings` fails on `ghui-app/src/telemetry.rs:188` and `ghui-app/src/updater.rs:90` with `clippy::uninlined_format_args` on the worktree's local toolchain. CI on Windows passes. When clippy fails with these two errors and nothing else, verify by stashing your changes and re-running on the parent commit — if the same two errors reproduce, they're not yours to fix in your PR. Document in PR body.
- 📌 Task 7 (Ghost-row visuals + click routing, 2026-05-19, PR #76): pretty smooth — the data side (`Node.isGhost`) was already wired by Task 2, so this was purely frontend. Two notable design moments. (1) The plan says "modify `WorkItemTree.svelte`" but the row rendering lives in the generic `TreeTable.svelte`; ghost styling and click handling went there, while the ghost-aware *menu* lives in `WorkItemTree.svelte` because that's where `MenuItem[]` is built. (2) An early helper draft had `resolveGhostClick(rows, clickedRowId)` that tried to look up the clicked row by id and then walk to the primary — but ghost and primary share an id (both are the WorkItemId), so `rows.find(r => r.id === clickedId)` always returned the primary and the test failed immediately. Caller already has the row object → drop the broken indirection, just use `findPrimaryRow`. Lesson: when extracting "pure" helpers, make sure the inputs actually carry enough information to be unambiguous. Helper module pattern (`ghostRouting.ts` + `.test.ts`) mirrors `filterableFields` / `recipeBarState` as expected. Follow-ups for Rusty: (a) auto-expand collapsed ancestors when jumping to a primary that isn't currently in the DOM — currently a debug-log no-op; (b) brief visual flash on the jumped-to row would be a nice touch but I didn't want to invent per-row selection state for it. Validation: 4 of 5 green; clippy fails on the same 2 pre-existing `uninlined_format_args` errors documented above (untouched Rust).

- 📌 Task 9 (Orthogonal toolbar toggles, 2026-05-19, PR #77): closes Phase 4 and the whole pivoting plan. Five-toggle toolbar — three live on three different state layers, and being deliberate about *which* layer matters more than how to wire the checkbox. (a) ``explodeMulti`` + ``showGhostAncestors`` → ``PivotConfig`` (persisted, already there). (b) ``hideClosed`` → new ``Filters.hide_closed: bool`` field in Rust with ``#[serde(default)]`` for cache backcompat. (c) ``showCounts`` + ``collapseSingleValue`` → ``$state`` on ``WorkItemContext``, frontend-only, intentionally NOT persisted because they are pure render concerns and the user is unlikely to want them sticky across restart. Three lessons. (1) When adding a non-field boolean to ``Filters``, the existing ``filterableFields`` iteration (which assumed ``keyof Filters`` = filterable single-select fields) breaks — introduce ``FilterableField = Exclude<keyof Filters, "hideClosed">`` *and* a runtime ``NON_FIELD_FILTER_KEYS`` set because TS narrowing doesn't survive ``Object.keys()``. Future maintainer must update both when adding more boolean toggles. (2) On the DelayLoad pattern: when filtering by ``is_closed()``, only filter on ``Loaded(true)`` — ``NotLoaded`` items must stay visible because dropping rows based on unknown data is a bug. Pinned in test ``_keeps_items_with_unloaded_state``. (3) ``PullRequest::is_closed()`` returns true for BOTH ``CLOSED`` and ``MERGED`` states (verified in test). The pre-existing top-bar "Hide Closed" button that hacked on ``filters.status`` is now rewired to drive ``Filters.hideClosed`` directly — same UI, more correct semantics (catches MERGED PRs even when the project Status field isn't "Closed"). Group-count derivation for ``showCounts``/``collapseSingleValue`` walks the flat depth-first ``data.nodes`` array — for each ``group`` head, count following ``workItem`` rows until level drops back to ``<= head.level``. Simple, O(n²) worst case but n is small in practice and there's no recursion or tree traversal needed. Validation: 4 of 5 green; same 2 pre-existing clippy errors in ``telemetry.rs:188``/``updater.rs:90`` (zero diff vs main). 27 vitest tests in ``RecipeBar.test.ts`` (was 14 — 13 new), 48 Rust tests in ghui-app (5 new for ``hide_closed``).
