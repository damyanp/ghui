## Learnings

<!-- Append new learnings below. Each entry is something lasting about the project. -->

- 📌 Svelte 5 runes, NOT Svelte 4 stores. `$state`, `$derived`, `$bindable`, `$props`.
- 📌 Context pattern: `setWorkItemContext()` in root `+page.svelte`, consumed in children. WorkItemContext lives in `app/src/lib/WorkItemContext.svelte.ts`.
- 📌 Vitest tests live next to source as `*.test.ts`. Plain TypeScript, no Svelte/Tauri runtime imports — `invoke` from `@tauri-apps/api/core` is NOT available in tests.
- 📌 Testable logic pattern: extract from `.svelte.ts` class → sibling pure module → class delegates. Example: `WorkItemContext` consumes `filterableFields.ts`.
- 📌 Styling: Tailwind 4 + Skeleton (Cerberus theme). Reuse tokens like `bg-primary-100-900`, `bg-surface-300-700`. NEVER add CSS comments with specific hex values — they go stale.
- 📌 Drag interactions: Pointer Events with `setPointerCapture` / `releasePointerCapture`, element-level handlers — NOT document-level (leak risk).
- 📌 Toolbar: all buttons via `AppBarButton`, always visible, disabled when unavailable.
- 📌 Icons: `@lucide/svelte` for UI, `@primer/octicons` for GitHub-specific.
- 📌 Bindings dir: `app/src/lib/bindings/` (auto-generated by ts-rs on `cargo test --all`). Don't hand-edit.
- 📌 Pivoting Task 3 component API: `<RecipeBar bind:value={cfg} onApply={(cfg) => …} />`. Props: `value: PivotConfig`, `onApply: (cfg: PivotConfig) => void`.
- 📌 Phase 2 additive rule: NEVER modify `WorkItemTree.svelte` rendering logic — reserved for Task 6. RecipeBar is built and tested standalone in `/dev/recipe-bar` demo route, NOT mounted in main toolbar until Task 6.
- 📌 Team update (2026-05-19): PR #71 review revealed dead-stub UI toggles ("Show counts", "Collapse single-valued", "Hide closed items") that are disabled with no backing behavior. These violate the project rule: *"Don't add UI elements until they have working functionality."* Linus: these stubs belong to Task 9 (final polish). Reject them from Task 3; leave disabled UI for later phases.
- 📌 Async parser pattern (2026-05-19, PR #71 revision): `recipeParser.ts` (merged from PR #73) is the ONLY recipe parser — a thin async Tauri shim. When refactoring components from sync to async parsing: (1) use `$effect` with a cancellation guard (`let cancelled = false; return () => { cancelled = true; }`) for external value sync; (2) track the last-emitted recipe reference (plain `let`, not `$state`) to skip redundant re-formatting on self-emitted updates; (3) extract `applyText()` and `setToggle()` pure helpers into a sibling `.ts` module so tests can inject mock parse/format functions without mocking `invoke` directly.
- 📌 RecipeBar prop interface (2026-05-19, PR #71 / Task 6): `<RecipeBar bind:value={cfg} onApply={(cfg) => …} />` — props are `value: PivotConfig` (bindable) and `onApply: (cfg: PivotConfig) => void`. Mount pattern for toggled panels: `$state(false)` boolean + `{#if open}<div class="border-b border-surface-300-700 px-4 py-2">…</div>{/if}` placed immediately after `</AppBar>`, before any panel components (same as LogPanel pattern).
